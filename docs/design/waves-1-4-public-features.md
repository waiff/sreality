# Waves 1–4 — public feature designs

**Status:** design/planning. The four public feature waves that ship *after* Phase 0 (hardening) and Phase 1 (multi-tenant foundations). Each was designed to implementation grade and adversarially reviewed; the corrected designs are below. Each wave ends with a re-audit (and, for Wave 4, legal) gate.

**Read first:** `phase-1-multitenancy-foundations.md` — every wave builds on its accounts/RLS/tenant-pool/entitlements foundation, and the **Amendments (A1–A8)** section there was written *from* these wave reviews. Cross-references below point at those amendments.

## Wave 1 — Chrome extension + agent-run rent estimations

**First public, first paid, the metered flagship.** The delta is thin because Phase 1 supplies every primitive (auth, `verify_jwt`, accounts, `current_account_ids()`, tenant pool, `usage_ledger`, `check_budget`, `entitlements`, dual-auth window). The wave is three things: give the extension a real per-user session, meter the agent estimation safely, and get through Chrome Web Store review.

> **Review correction — the load-bearing fix: authentication ≠ authorization.** The first draft swapped `require_token → verify_jwt` on the estimation/lookup routes but kept them on the **service-role** `get_db_conn`. That authenticates the caller but does **not** scope the row: `GET/PATCH /estimations/{id}` become **IDOR** (`run_id` is an enumerable `bigserial`; any tenant reads/overwrites any other's run, trace, and comparables), and `/listings/lookup` returns **account #1's** collections and pipeline placements to every tenant. These handlers must run on the **Phase 1 tenant pool** (RLS-enforced), exactly like Wave 2 — not merely be JWT-authenticated. This is the blocker; everything below assumes it's fixed.

**Extension auth — the extension owns its OWN Supabase session (not an SPA handoff).**
- Sign-in via `chrome.identity.launchWebAuthFlow` + **PKCE** against GoTrue (`redirect_to=https://<ext-id>.chromiumapp.org/`), exchange the code for `{access, refresh, expires_at}` in `chrome.storage.local`. **Never reuse the SPA's refresh token** — Supabase rotates with reuse-detection, so a shared token would lock out both sessions.
- Refresh: MV3 service-worker eviction kills `supabase-js` auto-refresh, so refresh **lazily inside the fetch wrapper** when near expiry **plus** a `chrome.alarms` periodic (~30 min); persist the rotated refresh token each time. `chrome.storage.local` is unencrypted → mitigate with a short access-TTL, rotation, and server-side revocation on sign-out.
- **Delete the build-time `VITE_API_TOKEN`** (`chrome-extension/src/api.ts:36,57`); the per-user JWT rides in `Authorization: Bearer` into the existing header gate (`allow_credentials=False` is untouched — the SW fetch runs with host permission and bypasses CORS; the JWT is a header, not a cookie).

**Metered agent estimations — the account_id hand-off, and execution off the request process.** The run's `account_id` **cannot** come from `auth.uid()` (the run persists in a service-role, no-JWT context), so it's **hand-threaded** from the verified request claims onto the run and onto one `usage_ledger(account_id, 'agent_estimation', cost_usd, now())` row at terminal (`cost_usd` = the run's `llm_calls` sum). This is Phase 1 **A4**; cost auto-attributes via `llm_calls.estimation_run_id` — `llm_calls` needs no `account_id`. **But the first draft's in-process post-response `BackgroundTask` is the wrong executor** (review): a 240 s agent run blocks one of Starlette's ~40 threadpool tokens the whole time, so ~20 concurrent paid runs stall the *entire* API (all sync routes share the pool); and every merge-to-`main` deploy SIGTERMs the container mid-run, killing paid runs with no resume and no ledger row. **Execution must drain off the request process** — a job lane on the already-always-on realtime worker consuming `pending` `estimation_runs` (the run row is still the job; only the *executor* moves). See Phase 1 **A10**. (The DB pooler is *not* the constraint here — the run uses short autocommit transactions between LLM calls — so don't over-index on pooling; the threadpool is.)

**Also migrate ALL extension-touched routes, not just estimation+lookup** (review): the extension also calls `/pipeline/cards`, `/collections/:id/properties`, `/properties/:id/notes` (`api.ts:141-238`). If the extension stops sending the static token but those routes still expect it, they **401**. Migrate every extension-touched route to `verify_jwt` + the tenant pool in Wave 1 (they're the same per-account writes Wave 2 moves anyway).

**Submit-time gates (before any spend) — and they must be ATOMIC** (review, Phase 1 **A9**): `require_entitlement('agent_estimations')` (server authoritative on `mode` — a client can't self-upgrade to `agent`); a per-account budget check; a per-account concurrency cap; and idempotency. The draft's `count-then-insert` / `select-then-insert` versions are all **TOCTOU-bypassable by parallel submits** (session advisory locks are unsound over the tx pooler — the mig-279 lesson). Implement as a **`UNIQUE` partial index `(account_id, target, kind) WHERE status IN ('pending','running')` + `ON CONFLICT`** (idempotency + concurrency in one atomic write) and an atomic `INSERT … SELECT WHERE (window count) < limit` for budget. **Meter parse cost too:** `_resolve_input` runs the LLM URL parse *before* the run row exists (`estimation_runs.py:475`), so it's invisible to a ledger written at terminal — count/authorize it up front, and place the idempotency short-circuit **before** the parse, not after. **Block both bypasses:** reject raw `spec` *and* `spec_overrides` (`schemas.py:341`) for extension/non-admin callers (both drive arbitrary-lat/lng agent searches). Note the real spend vector is the agent's comparable-search iterations, not the parse — so the budget/concurrency caps, not the spec block, are the primary control. The enforcing daily kill-switch is Phase 1 work — today's `_check_daily_cost` is warn-only, global, and post-hoc, so don't cite it as the per-account backstop (and a *global* cap means one abuser DoSes everyone — it must be per-account).

**Zombie-run hygiene** (review): the stuck-run sweep is **startup-only** (`api/main.py:117`), so an orphaned `running` row permanently consumes a concurrency slot and poisons idempotency (the user polls a corpse forever). Add a **periodic** sweep (pg_cron or the worker lane), and make the cap/idempotency treat stale rows as free.

**Async submit→poll (already the model, but under-tuned).** Widen the poll window past the 240 s wall clock — `60×2000ms = 120s` (`content.ts:35-36`) reports a succeeding run as "failed"; raise to >300 s or derive from the run's skill limits — and **persist the in-flight `run_id` to `chrome.storage.local`** so a re-open resumes instead of re-submitting. Canonicalize the URL for the idempotency key (query-param/scheme/trailing-slash variance across portals otherwise double-charges; `sreality_id` keys are clean). The extension must send `mode:'agent'` for entitled users (today it hard-codes `deterministic`, `api.ts:128`) and the panel must distinguish a server-downgraded free result from a paid one.

**Chrome Web Store readiness:** `host_permissions` narrows to the **API origin *and* the Supabase origin** (the SW fetches `${SUPABASE_URL}/auth/v1/token` for PKCE exchange + refresh — "just the API origin" breaks auth); add `identity` + `alarms`; pin a stable extension ID via a manifest `key`, and register the `https://<ext-id>.chromiumapp.org/` redirect in **both** Supabase `additional_redirect_urls` **and** the Google OAuth client (the config that actually fails in practice); `sourcemap:false` + `minify:'esbuild'`; privacy policy + per-permission justifications + single-purpose statement; **staged rollout** with the API backward-compatible across ≥2 versions.

**Token rotation needs a concrete cutoff, not "when old builds age out"** (review): `VITE_API_TOKEN` *is* the shared admin god-token, already sitting in cleartext in every shipped bundle. Deleting it from *future* builds changes nothing until the value is **rotated** — and rotation is platform-wide (it also breaks the SPA and ClickUp), so sequence it as an explicit dated cutoff after a forced-min-version check, not an open-ended deferral.

**Schema delta (thin):** add `'extension'` to `estimation_runs.source` (CHECK migration + the Pydantic `Literal` in lockstep — the SQL-correctness CI gate enforces it); the `UNIQUE` partial concurrency/idempotency index; seed the metered action + per-plan quota into `entitlements.features`. No new job table (the run row is the job). No new deps (`chrome.*` platform APIs; GoTrue PKCE is a hand-rolled fetch — no `supabase-js` in the vanilla-TS extension).

**Compliance — corrected target** (review): `/listings/lookup` already carries **no** broker contact fields, so redacting it is a no-op; the real leaks are (a) the cross-tenant curation state from the IDOR fix above and (b) free-text broker contacts an agent could embed in `trace`/`comparables_used` — redact *those*. Account deletion **anonymizes** immutable estimation runs (rule 12) rather than deleting. The plaintext long-lived refresh token in `chrome.storage.local` is an accepted residual risk of the own-session choice (mitigate with short access-TTL + rotation + server revoke; single-flight the lazy + alarm refresh paths so reuse-detection doesn't log the user out). **Launch gate:** RLS test lane green + external re-audit + a two-real-account pen-test asserting A can't read B's runs (the IDOR), the quota/concurrency caps actually reject under parallel submits, and no broker PII in any extension response. **Open decisions:** metered unit (recommend per-successful-run against a monthly quota, cost recorded for margin); free tier (deterministic free, agent metered); charge policy on partial/failed-but-costly runs (recommend absorb the count, record the cost).

## Wave 2 — Opportunity pipeline management

**The thesis: this is a connection-swap, not a rewrite.** Phase 1 already re-keyed `property_pipeline` to `(account_id, property_id)` and account-scoped `pipeline_stages`. So the eight `/pipeline/*` routes plus `POST /listings/lookup` mostly just move from the service-role pool to the **tenant pool** (`get_tenant_conn`), and RLS then scopes every global scan — the entry-stage lookup (`api/pipeline.py:176`), the crown-move that clears `is_entry` on the other stage (`:79-83`), the reorder live-set, `max(board_position)` — to the caller's account automatically, failing closed if a predicate is forgotten. The SPA board reads (`fetchPipelineBoard`, `fetchPipelineMemberSet`, `fetchPropertyPipeline`, `fetchPipelineStages`) need no logic change once they run as `authenticated` through the `security_invoker` views. The `<FunnelIcon>` affordance on Browse cards, the listing header, the kanban, and the extension becomes per-account for free.

**What's genuinely new (three small, low-risk pieces):**
- **Cross-account stage-ownership FK** — the one hole RLS can't plug. A client can PATCH `MoveCardIn.stage_id` with *another* tenant's stage id; the single-column FK (`migrations/205:36 → pipeline_stages(id)`) is a global check that accepts it, leaving account A's card pointing at account B's stage (leaks B's label/color through `property_pipeline_public`). Fix: `unique (account_id, id)` on `pipeline_stages`, drop the single-column FK, add composite FK `property_pipeline(account_id, stage_id) → pipeline_stages(account_id, id)`. A cross-account id now raises `ForeignKeyViolation`, which `move_card` already catches into a no-info-leak 422.
- **`seed_default_pipeline(account_id)`** — a `SECURITY DEFINER` function copying the five default stages per account, called inside Phase 1's signup transaction so a new user always has an entry stage before their first bookmark. Hardcode the five rows in the function body (skip a templates table unless the operator wants runtime-editable defaults). **Depends on Phase 1 Amendment A3** — `pipeline_stages_key_ci` must be re-keyed `(account_id, lower(key))` or the *second* signup's seed collides.
- **Row-local `CHECK (not (is_entry and is_terminal))`** + an account-leading board index `(account_id, stage_id, board_position)`.

**Corrections folded in from review:**
- **The merge reconciler is the real work, not a "verification"** (Phase 1 **A2**). `reconcile_pipeline_on_merge` runs `BYPASSRLS`; with N cards per property it must be rewritten account-partitioned (`s.account_id = r.account_id` everywhere; snapshot/restore per `(account_id, property_id)`), and `property_pipeline_events` must gain `account_id`. Property merges are market-wide and continuous, so getting this wrong is routine cross-tenant corruption. This is Phase 1 pipeline-rewrite scope, called out here because Wave 2 can't ship correct without it.
- **`board_position` needs a real concurrency fix** (accounts have members from day one): two members dropping into the same column read the same `max()+1`. Use a per-`(account, stage)` advisory lock or `unique (account, stage, board_position)` + retry; keep the `added_at` secondary sort as *defense-in-depth*, not the fix.
- **`portal_lookup` on the tenant pool** scopes its `property_pipeline` + `collection_properties` joins by RLS — but that requires Phase 1 **A5** (explicit `authenticated` read policies on the shared base tables it reads directly, `listings`/`properties`), or the endpoint returns nothing.
- **`estimation_runs` visibility** in `portal_lookup`'s LATERAL estimate join needs Phase 1 **A4** (nullable `account_id`, policy `account_id IN current_account_ids() OR = SYSTEM`) so the platform golden estimate still surfaces.
- **Naming hygiene:** the *dedup* pipeline (`DedupFunnel`, `pipeline_checks_public`, `pipeline_overview`) is a different domain — admin-gated, service-role, never per-account. Keep it cleanly separate.

**Deploy note:** the pipeline WRITE routes hard-fail under the tenant pool without a user JWT, so extension pipeline writes can't ship until Wave 1's `chrome.identity` session lands (a real sequencing constraint). No new dependencies (`@dnd-kit` is already bundled). **Launch gate:** the RLS test lane green for every pipeline surface + a `/code-review ultra` re-audit + a manual two-account pen-test (A cannot see/move/archive/reorder/inject into B's board via SPA, API, or extension; a fresh signup lands a private five-stage board). **Compliance:** the board is greenfield per-account PII — register `property_pipeline` + `property_pipeline_events` in Phase 1's deletion cascade/export (this ledger is *user state*, purged on deletion, **not** sacred market history); the card's broker-contact hover inherits Wave 4's masking (don't hardcode raw contact rendering).

## Wave 3 — Watchdogs & notifications

**Detection is broken for a public product on three axes today** and the fix is mostly architectural. (1) The matcher runs as an asyncio task **inside every FastAPI replica** (`api/main.py:144-147`) — N replicas fan out N× duplicated matching with no single-runner guard. (2) `match_once` window-scans `properties_public` on `l.published_at` with **no btree index** on `properties.published_at` (the column exists since migration 273; only a partial gate-expression index was ever built) — O(subs × ~53k) per pass. (3) Subscriptions have no owner.

**The corrected detection design (review pruned the over-engineering):**
- **Ship the index** — `CREATE INDEX CONCURRENTLY properties_published_at_idx ON properties (published_at) WHERE published_at IS NOT NULL`. This is the single highest-value fix; keep the existing **per-subscription cursor scans** (each bounded by its own `published_at > cursor` + `LIMIT window`).
- **Drop the shared-delta materialization** the first draft proposed — a "materialize once, JOIN per sub" TEMP TABLE **strands on the transaction pooler** (rebinding backends — the exact bug migration 279 was written for) and couples every pass to the single laggard cursor. The index alone is the robust fix.
- **Move the three producer passes to a leased lane on the always-on realtime worker**, guarded by a `notification_matcher_lease` CAS row (migration 279 shape — pooler-proof, unlike advisory locks). But **keep a leased pass co-hostable in the API** (Phase 1 **A8**) so detection isn't coupled to the dark-by-default worker; or gate `REALTIME_WORKER_ENABLED` with a stale-heartbeat alarm.
- **Fix the pre-existing city-quality/geom lockstep bug first:** `_city_quality_clauses` emits `l.geom` predicates, but `properties_public` projects only lat/lng (no geom), so any city-index or proximity watchdog throws `column l.geom does not exist`, swallowed by the per-sub try/except → **silently never matches**. Rewrite those branches to lat/lng (as the radius branch already does) before claiming rule #16 lockstep.

**Ownership + RLS:** `notification_subscriptions`, `notification_dispatches` gain `account_id NOT NULL` (denormalized from the owning subscription/collection; `system_health` rows → SYSTEM account) + RLS; the user-facing feed/unread/mark-seen/estimate routes move to the tenant pool. **`channel_sends` does NOT get `account_id`+RLS** (Phase 1 **A7**) — it's a shared ledger across four consumers incl. the GDPR-global broker-outreach path; derive per-account delivery status by joining through `notification_dispatches.account_id`. `seen_at` read-state stays per-account (correct once subscriptions are owned).

**Delivery** (the ledger + Resend/Telegram transports are already built but have never run — recipient resolves from *one* global `app_settings` pair):
- Per-account recipients in a new `account_notification_settings` (email + telegram + digest mode + tz), gated by **double-opt-in verification** (mandatory before any send — the outbox hard-skips unverified).
- **Global-by-address suppression** (`notification_suppression(channel, address)`, mirroring `broker_outreach_suppression`) with a pre-send gate; a bounce/complaint/unsubscribe suppresses that address everywhere and survives tenant deletion.
- **RFC 8058 one-click List-Unsubscribe** header on every email + an unauthenticated HMAC-verified `/u/{token}` endpoint that renders for logged-out users.
- **Signature-verified Resend bounce/complaint webhook** (Svix HMAC via stdlib `hmac`/`hashlib` — **no new dependency**; same auth class as the Phase 1 Stripe webhook) feeding suppression and widening `channel_sends.status` to `delivered/bounced/complained` (foreseen at migration 207).
- Optional per-account **digests** (needs an atomic multi-dispatch claim + a `digest_key` group in the outbox — specify the claim/finalize semantics or you get immediate+digest double-sends).

**Quotas** (per Stripe entitlements): max active watchdogs (enforced at create/activate), channel entitlements, and priority detection cadence (a per-tier `last_evaluated_at` gate inside the single pass). **The daily-dispatch cap goes on DELIVERY, not detection** (review): capping at the matcher INSERT permanently loses matches (the cursor advances past dropped rows and the once-ever `dedupe_key` never re-fires, violating rule #16 that in-app *is* the row) — instead leave in-app dispatches uncapped and cap emails/telegrams in the outbox by deferring (the dispatch row persists).

**New env (all backend-only):** `RESEND_WEBHOOK_SECRET`, `NOTIFICATION_UNSUB_SECRET`. **Cross-tree imports into the worker must be lazy** (top-level `from api…` in the worker is a silent boot crash — the API-import-surface rule). **Launch gate:** cross-tenant audit of the three tables + webhook + unsubscribe; GDPR sign-off on double-opt-in / one-click unsubscribe / global suppression / Resend US residency (transactional-only); a deliverability check (verified domain, bounce webhook firing); and a load check that one matcher pass at target sub count stays within the worker interval.

## Wave 4 — Broker analytics

**Technically small, legally the hardest — the legal sign-off *is* the launch gate.** Today the broker leaderboard/detail ranks ~30k identifiable persons and publishes their `primary_email`/`primary_phone` as `tel:`/`mailto:` cards, flowing through anon-granted `brokers_public` / `broker_leaderboard()` that the SPA reads directly. Scraped `rating`/`review_count` are *stored* (`migrations/185:100-101`) but never surfaced.

**The critical sequencing correction (review, and Phase 1 A6):** the naive Phase 1 blanket anon→authenticated re-grant would expose `brokers_public` *with contacts* to every tenant **before** this wave masks anything. So Phase 1 must **exclude every broker-PII surface from the re-grant** (they stay dark to tenants), and **Wave 4 is the sole gate that flips them to `authenticated` in masked form** — in the same PR as the masking re-sign, never before. Encode this as a CI assertion.

**Buildable now (no legal blocker to writing the code, ship it dark behind the gate):**
- **Masking re-sign** (`mig 286`, DROP+CREATE per the mig-190 apply-and-merge-promptly discipline): `brokers_public` and `broker_leaderboard()` stop returning `primary_email`/`primary_phone` (add `has_email`/`has_phone` boolean flags at most). Raw contacts stay strictly behind the `require_admin` service-role API.
- **`get_broker` must be decomposed, not just re-signed** (review): it inlines the full PII ledger via `_contacts()` reading base `broker_identity_contacts` (service-role, RLS-bypassing) — masking the *view* does nothing for that path. Split the contact portion behind `require_admin`; repoint the admin header read to base `brokers`.
- **Unify suppression** (review): one `broker_suppression(broker_id, scope in ('display','outreach'))` (or a `display_suppressed` flag on the existing outreach table), **not** a second parallel do-not-X table — one anti-join, one test contract. Both scopes stay **GLOBAL** (a display/outreach opt-out applies across all tenants).
- **Complete the suppression surface list** (review): the anti-join must cover *every* `broker_id`-bearing surface — `brokers_public`, `broker_leaderboard`, `broker_listings_public`, `listing_broker_public`, **`broker_firm_memberships_public`** (re-identifies by broker_id→firm+dates), and the directly-granted **`broker_region_type_stats` matview** (revoke its direct grant and force reads through a filtered view, else a tenant reads a suppressed broker_id straight from the matview). Decide semantics = **exclude the row** (blank-name still ranks and re-identifies via click-through). One test enumerates all surfaces and asserts a suppressed broker returns zero rows everywhere.
- **Fix the outreach-targeting join, don't clone it** (review): `select_targets` filters on `broker_leaderboard(...).primary_email`, which the masking removes → a hard SQL break the schema-aware CI gate will catch. Keep ONE contactless leaderboard for ranking and JOIN broker email by `broker_id` in the service-role admin path — do **not** clone the heavy group-by into a `broker_targets_admin()` (re-risks the mig-187 distinct-count footgun).
- **Firm-level default is the wave's *largest* build, not "defensive"** (review): a firm leaderboard driven by the Browse region picker needs a **new `firm_region_type_stats` matview** with per-level distinct counts (`firms_public` has only totals; `broker_region_type_stats` is broker-grain with no `firm_id`). Budget it honestly, and beware the distinct-count-per-geo-level footgun mig 187 already warns about.
- **Redaction rationale fix** (review): `review_count`/`review_score` are **excluded** from the content hash (`VOLATILE_PREMISE_KEYS`), so a forward NULL-migration on `broker_identities.rating` causes **zero** snapshot churn — trivial if legal mandates purge (never a destructive delete; "rows immortal, personal fields redactable").
- **The public opt-out endpoint fights the architecture** (review) — it re-opens an anon write path Phase 0 just closed and needs a broker-impersonation-proofing subsystem. Instead: a privacy-page `mailto:`/DPO inbox that an **admin** processes into the suppression table (write stays behind the existing perimeter; no new auth class). Revisit self-service only if volume demands.

**Blocked on legal (the gate):** the DPIA (Art 35 large-scale systematic profiling), the lawful-basis/LIA, whether raw contacts may *ever* reach a tenant (recommend firm-level default + raw admin-only), keep-vs-purge scraped ratings, the sui-generis DB right + portal ToS/robots review for re-publishing scraped identities to third parties, the Art 14 approach (likely a public privacy notice under the disproportionate-effort exemption), and **Act 480/2004 opt-in before ANY per-tenant outreach** — so outreach **stays admin-only this wave** (it's already paused). Per-tenant outreach (add `account_id`+RLS to campaigns/messages, `outreach_sender_identities`, per-account consent, per-send suppression re-check, an LLM hard spend cap) is a **separate legally-gated sub-wave**. **Extension: no change** (it surfaces zero broker data). **Launch gate = legal sign-off + the mandatory Wave-4 broker-PII re-audit.**

## Cross-wave dependency map

```
Phase 0 (hardening)  ──►  Phase 1 (auth, accounts, RLS, tenant pool, entitlements, admin gating)
                                │  (incl. Amendments A1–A8, several of which THESE waves surfaced)
        ┌───────────────┬───────┴───────┬────────────────┐
     Wave 1          Wave 2           Wave 3           Wave 4
   extension +     pipeline         watchdogs +      broker analytics
   agent est.      (needs W1        notifications    (needs Phase 1 A6:
   (metered;       session for      (needs A4, A7,   broker PII excluded
   needs A4)       ext. writes;     A8; matcher      from the re-grant;
                   needs A2, A3,    lease + index)   legal gate)
                   A5)
```

**Ordering constraints:** Wave 2's extension pipeline writes depend on Wave 1's per-user session. All waves depend on the Phase 1 amendments (A1–A8). Wave 4's masking must land in the same window Phase 1 would otherwise expose broker PII — so A6 (exclude broker-PII from the re-grant) is a **hard prerequisite baked into Phase 1**, and Wave 4 is the only gate that flips those surfaces on (masked).
